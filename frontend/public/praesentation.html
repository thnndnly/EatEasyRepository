Problem & Idee

„Was kochen wir diese Woche — und was fehlt dafür?"

Mahlzeitenplanung im Haushalt ist verteiltes Wissen: Rezepte im Kopf, Vorrat im Kühlschrank, Einkauf auf einem Zettel. EatEasy führt das an einem Ort zusammen.

  • Gemeinsam statt einzeln — ein Haushalt, mehrere Mitglieder, geteilte Rezepte & Pläne.
  • Kein doppelter Einkauf — die Liste entsteht automatisch aus Plan minus Vorrat.
  • Weniger Denkarbeit — die KI schlägt Gerichte vor, die der Vorrat schon fast hergibt.

01

Die Architektur

Wie EatEasy geschnitten ist — und warum genau so.

~8 Minuten
Grundentscheidung

Modularer Monolith, kein Microservice-Zoo

  • Fachliche Module statt technischer Layer — geschnitten nach Domäne, nicht nach „alle Controller / alle Services".
  • Ein Deploy, klare Grenzen — die Trennschärfe von Services, ohne Netzwerk-, Latenz- und Betriebs-Overhead.
  • Herauslösbar geschnitten — die Komponentengrenzen bleiben ein späterer Service-Split-Pfad.
Leitregel

Komponenten reden über Service-Interfaces — nie über REST, nie über fremde Repositories.

Wäre der Schnitt nötig

Auth · Core · Smart-Suggestion wären die natürlichen Service-Splits. Der Code ist so strukturiert, dass das möglich bliebe.

Komponenten-Landkarte

Zwölf Komponenten, drei Sorten

142 Java-Dateien. Jede fachliche Komponente exponiert genau ein Service-Interface nach außen.

Schichten innerhalb einer Komponente

Immer dieselbe Ordnung — Beispiel recipe

resource/JAX-RS-Endpunkte unter /api/v1/recipes — dünn, delegiert sofort. @RolesAllowed("user")
dto/Request/Response als Java Records mit Bean-Validation — Entities lecken nie nach außen.
service/RecipeService (Interface) + RecipeServiceImpl — Business-Logik, Autorisierung, @Transactional.
repository/Panache-Repositories — RecipeRepository, kompakte Queries.
entity/JPA-Entitäten — Recipe, RecipeIngredient, RecipeFavorite.

Andere Komponenten sehen nur das Service-Interface. Alles darunter ist privat.

Die wichtigste Regel

Service-Interface rein, Repository bleibt privat

Braucht die Einkaufsliste den Vorrat, injiziert sie PantryService — niemals PantryRepository. So bleibt jede Komponente Herr über ihre eigenen Daten und Auth-Regeln.

Sicherheit

Autorisierung im Service-Layer — nicht nur per Annotation

household/service/HouseholdService.javaZ. 65
// Default-Method im Interface: ein Guard,
// den jede fremde Komponente wiederverwendet.
default void assertMember(
        UUID userId, UUID householdId) {
    if (!isMember(userId, householdId)) {
        throw new ForbiddenException(
            "Kein Zugriff auf diesen Haushalt");
    }
}
  • @RolesAllowed heißt nur eingeloggt — nicht berechtigt für genau diese Ressource.
  • Jeder Service prüft zusätzlich: gehört die Ressource zu meinem Haushalt?
  • Ein Guard im Interface — assertMember / assertOwner —, den alle Komponenten teilen.
  • Auch Smart-Suggestion & Einkaufsliste rufen zuerst assertMember.

Zwei Wächter: Annotation (authentifiziert) + Service-Check (autorisiert).

Frontend-Spiegelung

Das Frontend spiegelt die Backend-Komponenten 1:1

  • Ein Pinia-Store je Komponente — useRecipeStore, usePantryStore … Server-State lebt nur dort.
  • Klare Schichten — View → Store (State + loading/error) → Service → Backend.
  • Der Token wird durchgereicht, nie im Service gehalten — zentrale Bearer-Injektion im Wrapper.
  • Lazy-loaded Views, deklarative Route-Guards (requiresAuth / guestOnly).

Station 2 · Herzstück

Einkaufsliste = skalieren, normalisieren, aggregieren

shoppinglist/…/ShoppingListServiceImpl.java · aggregateNeeded()Z. 169–178
BigDecimal scaling = BigDecimal.valueOf(entry.servings())
    .divide(BigDecimal.valueOf(mini.servings()), SCALING_PRECISION, RoundingMode.HALF_UP);
for (RecipeIngredientView ingredient : ingredients) {
    BigDecimal scaled = ingredient.amount().multiply(scaling);
    // gleiche Zutat in verschiedenen Einheiten (TBSP vs ML) auf
    // eine kanonische Einheit bringen, damit sie zusammenfällt:
    Unit canonicalUnit   = UnitConverter.canonical(ingredient.unit());
    BigDecimal canonical = UnitConverter.toCanonical(scaled, ingredient.unit());
    needed.merge(new Key(ingredient.ingredientId(), canonicalUnit),
                 canonical, BigDecimal::add);
}

BigDecimal statt double (Mengen-Genauigkeit) · merge summiert pro (Zutat, Einheit) · danach wird der Vorrat mit derselben Kanonisierung abgezogen. Beim Neuberechnen bleibt der checked-Status je Zutat erhalten.

Station 3 · KI mit Netz

Erst rechnen, dann fragen — LLM nur für den Grund

suggestion/…/SmartSuggestionServiceImpl.javaZ. 90–113
// 1) deterministisch: Vorrats-Abdeckung je Rezept
Map<UUID,Double> coverage =
    CoverageCalculator.compute(ingredientsByRecipe, pantryIds);
List<Candidate> candidates = recipes.stream()
    .map(r -> new Candidate(r, coverage.get(r.id())))
    .filter(c -> c.coverage >= COVERAGE_THRESHOLD)  // 0.5
    .sorted(comparingDouble(c -> c.coverage).reversed())
    .limit(CANDIDATE_LIMIT).toList();              // max 10

// 2) LLM begründet nur — fällt es aus, bleibt die Reihenfolge
OllamaOutcome outcome = askOllama(candidates, pantryIds);
return new SuggestionResponse(outcome.aiAvailable(), list);
  • Heuristik trägt die Demo — Ranking ist reproduzierbar, auch ohne KI.
  • Drei Ausfälle abgefangen — leere Antwort, unparsbares JSON, Exception → aiAvailable=false.
  • Frontend zeigt den Modus — echte Begründung vs. „aus Vorratsabdeckung berechnet".
Station 4 · CDI-Feinheit

Ein Interface, zwei Provider — Auswahl per Config

Dieselbe Pipeline läuft lokal auf self-hosted Ollama und in der Cloud-Demo auf gehostetem Groq. Kein Code-Zweig im Service — CDI liefert die passende Implementierung.

Warum es zählt

Render hat kein RAM für ein lokales LLM. Eine Env-Variable — AI_PROVIDER=groq — genügt.

suggestion/client/LlmClientProducer.javaZ. 20–33
@Produces @ApplicationScoped
public OllamaClient llmClient(
    @ConfigProperty(name = "ai.provider",
        defaultValue = "ollama") String provider,
    OllamaHttpClient ollamaClient,
    GroqClient groqClient) {
  if ("groq".equalsIgnoreCase(provider.trim())) {
      return groqClient;
  }
  return ollamaClient;
}
Technik-Highlight · KI-Robustheit

Drei Antwortformen von einem Modell ohne Regeln

suggestion/…/SmartSuggestionServiceImpl.java · parseOllamaResponse()Z. 218–256
JsonNode root = objectMapper.readTree(body);
List<JsonNode> entries;
if (root.isArray()) {                        // 1) spec-konform
    entries = toList((ArrayNode) root);
} else if (root.isObject() && root.has("recipeId")) {
    entries = List.of(root);                 // 2) nur ein Objekt
} else {
    JsonNode arr = findFirstArray(root);     // 3) irgendwo im Wrapper
    entries = arr != null ? toList((ArrayNode) arr) : List.of();
}
for (JsonNode n : entries) {
    try { result.put(UUID.fromString(id(n)), reason(n)); }
    catch (IllegalArgumentException ignore) { /* halluzinierte ID */ }
}

Defensiv gegen ein nicht-deterministisches LLM: Array, Einzelobjekt oder Wrapper werden akzeptiert, eine rekursive Suche findet das erste Array im Baum, halluzinierte UUIDs werden still verworfen. Scheitert alles, greift der Coverage-Fallback (aiAvailable=false).

Technik-Highlight · Quarkus

Ein REST-Client ohne eine Zeile Implementierung

integration/client/TheMealDbClient.javaZ. 20–32
@RegisterRestClient(configKey = "themealdb")
@Path("/api/json/v1/1")
public interface TheMealDbClient {

    @GET @Path("/search.php")
    TheMealDbResponse search(@QueryParam("s") String q);

    @GET @Path("/lookup.php")
    TheMealDbResponse lookup(@QueryParam("i") String id);
}
  • Nur ein Interface — Quarkus generiert den HTTP-Client zur Build-Zeit aus den Annotationen.
  • Base-URL aus der Config via configKey, nicht im Code hartverdrahtet.
  • @QueryParam bindet direkt auf die Methodenparameter — kein manuelles URL-Bauen.
Technik-Highlight · Nebenläufigkeit

Doppelklick-sicher: ON CONFLICT DO NOTHING

recipe/repository/RecipeFavoriteRepository.javaZ. 28–37
public boolean insertIfAbsent(UUID userId, UUID recipeId) {
    int inserted = getEntityManager().createNativeQuery(
        "insert into user_favorite_recipe "
      + "(id, user_id, recipe_id, created_at) "
      + "values (?1, ?2, ?3, current_timestamp) "
      + "on conflict (user_id, recipe_id) do nothing")
        .setParameter(1, UUID.randomUUID())
        .setParameter(2, userId).setParameter(3, recipeId)
        .executeUpdate();
    return inserted > 0;   // true = neu, false = gab es schon
}
  • Atomar auf DB-Ebene statt check-then-act in zwei Schritten.
  • Schützt gegen die TOCTOU-Race bei Doppelklick oder zwei Tabs.
  • Der Rückgabewert trennt „neu angelegt" von „gab es schon" — ohne zweite Query.
Technik-Highlight · Sicherheit

Google-Login: Token prüfen, nicht dem Client glauben

  • Kein Google-SDK, kein OIDC-Server-Flow — nur der JDK-HttpClient gegen Googles tokeninfo.
  • Audience-Check clientId == aud verhindert Token-Substitution.
  • Mündet in denselben stateless-JWT-Flow — Ergebnis ist ein EatEasy-JWT.
auth/google/GoogleTokenVerifierImpl.javaZ. 61–95
var res = httpClient.send(request,
    HttpResponse.BodyHandlers.ofString());
if (res.statusCode() != 200)
    throw new InvalidCredentialsException();
JsonNode root = objectMapper.readTree(res.body());
String aud = root.path("aud").asText("");
// Schutz gegen Token-Substitution:
if (!clientId.equals(aud))
    throw new InvalidCredentialsException();
return new GoogleIdTokenPayload(
    text(root, "email"), text(root, "sub"),
    text(root, "name"));
Reflexion

Herausforderungen & Learnings

Aggregation

Einheiten sind tückisch

TBSP, ML, Gramm, Stück — nur mit kanonischer Normalisierung und BigDecimal stimmt die Einkaufsliste wirklich.

KI verlässlich machen

LLMs halluzinieren

Kleine Modelle liefern nicht immer valides JSON. Robustes Parsing + Coverage-Fallback statt Blind-Vertrauen.

Komponenten-Disziplin

Grenzen halten kostet Bewusstsein

Der einfache Weg wäre der direkte Repository-Zugriff. Service-Interfaces + Batch-Methoden halten den Schnitt sauber.

Betrieb im Free-Tier

Demo ≠ Prod

Kaltstart, kein OCR-RAM, gemockte Mails — Feature-Flags trennen Demo-Realität von voller lokaler Funktion.
